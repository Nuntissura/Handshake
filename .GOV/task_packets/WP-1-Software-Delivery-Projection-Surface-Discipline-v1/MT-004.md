- WEAK_SPOTS:
  - **Least-proven requirement**: the production writer derives lifecycle inputs from the *canonical structured-collaboration summary* + *canonical overlay records* — it is monotonic and idempotent over canonical truth. Future runtime engines that need to express lifecycle truth that canonical cannot yet carry (e.g., an explicit `workflow_failed` event from a flight recorder fault that has not yet propagated to `tracked_wp.status`) must either (a) extend canonical truth so the writer derives the same value, or (b) write the sidecar AFTER `finalize_runtime_structured_work_packet_writes` runs in their own code path. The existing override semantics in `compute_software_delivery_projection_gate_posture_from_disk` remain in place and are still tested at the projection-emitter level by `projection_emitter_respects_seeded_lifecycle_override_for_*`, but the production lifecycle writer would overwrite a divergent sidecar on the next finalize call.
  - **Riskiest file/boundary**: `apply_software_delivery_workflow_run_lifecycle` shares the canonical-record reader with `apply_software_delivery_projection_surface_lifecycle` (both call `read_canonical_software_delivery_overlay_records` + `select_software_delivery_projection_stable_ids` + `derive_software_delivery_projection_lifecycle_flags_from_canonical`). Stable ids on the lifecycle record are guaranteed to match the projection surface stable ids by construction, but a future divergence between these helpers could let the sidecar lag canonical without causing a compile error.
  - **Carry-forward**: `workflow_binding_state` derivation is tested for Validation/Review/Approval/Done/Failed/Canceled/Settled families through canonical inputs and overrides; Intake/Ready/Waiting/Blocked branches are exercised by code path but not in dedicated assertions in the new tests (would expand the test scope beyond MT-004 contract).
