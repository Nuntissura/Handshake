- NOTES:
  - MT-005 is a TRIPWIRE microtask; the mailbox advisory contract is already advisory-by-construction (the only software-delivery surface in role_mailbox.rs is `build_software_delivery_overlay_triage_row`, a pure function from projection surface to triage row). The new test pins the contract: it materializes the full canonical software-delivery overlay state on disk (claim/lease, queued instruction, `projection_surface.json`, `workflow_runs/<wp_id>.json`), builds the triage row, mutates the triage row in arbitrary ways (forged ids, settled binding state, swapped record refs, injected mailbox thread ids), serialises/deserialises the mutation, and asserts every canonical artefact is byte-identical to the pre-mutation snapshot. This proves a mailbox reader/UI cannot silently re-derive ownership, queued steering, or workflow lifecycle state from a triage row.
  - Authority-boundary discipline preserved: the triage row builder still refuses non-software_delivery profiles, and re-deriving the triage row from the unchanged projection produces an output equal to the original — the mutated copy never linked back to canonical state.
  - Legitimate channels for mailbox-driven authoritative mutations (governed action / transcription via `add_transcription_link`) are out of scope for MT-005; the test scope is the negative path ("a triage row mutation cannot mutate canonical state").
- WEAK_SPOTS:
  - **Least-proven requirement**: the test validates that the existing structural design is advisory; it does not statically prevent a *future* commit from adding a mutator method on `RoleMailbox` that takes a triage row and writes claim/lease, queued instructions, or workflow_run lifecycle records. A static lint or grep guardrail (e.g., a build-time check that no path in `role_mailbox.rs`/`api/role_mailbox.rs` calls `claim_lease_record_path`/`queued_instruction_record_path`/`workflow_run_record_path`/`work_packet_projection_surface_path`) would harden the boundary against future regressions.
  - **Riskiest file/boundary**: `api/role_mailbox.rs::add_transcription_link` is the legitimate channel for transcription-driven mutation of linked authoritative state. The test does not exercise that code path (out of scope for MT-005), so a future regression that quietly extended transcription targets to bypass governed-action validation would not be caught by this tripwire alone.
  - **Carry-forward**: the triage row currently carries no `mailbox_local_versus_governed_action` discriminator field for the per-row InteractionContract (Reply local vs. governed vs. transcription-required); refinement.md UX_AND_API_GUIDANCE describes this contract but it is not yet a structured field on `SoftwareDeliveryOverlayTriageRowV1`. Adding it would be additive and outside MT-005's clause scope.
