# Pi (pi-mono) — Deep Technical Research

Research target: `pi`, the coding agent inside Mario Zechner's `pi-mono` monorepo.
Local clone explored: `harnesses/pi-mono` (commit on local `main`, April 2026).
Author: Mario Zechner (`badlogic`). Repo: https://github.com/badlogic/pi-mono.

---

## 1. Executive summary

Pi is a single-process, single-loop ReAct-style coding agent built on top of a small layered stack the same author also publishes (`@mariozechner/pi-ai` for unified provider streaming, `@mariozechner/pi-agent-core` for the loop, `@mariozechner/pi-coding-agent` for the CLI). It deliberately ships with **four tools** (`read`, `bash`, `edit`, `write`), a sub-1k-token system prompt, **no MCP, no sub-agents, no plan mode, no permission popups, no built-in todos, no background bash**. Everything else — sub-agents, plan mode, permission gates, MCP clients, custom compaction — is either an extension that you load, an externally written CLI tool the model calls through `bash`, or a separate `pi` subprocess spawned via `bash`/`spawn()`. The harness's top-level design rule is that nothing should silently enter the model's context: "injecting stuff behind your back that isn't even surfaced in the UI" makes context engineering "extremely hard or impossible" (Zechner, blog, 2025-11-30). Pi is the antithesis of an artifact-driven multi-agent broker.

## 3. Architecture

Pi is a **single-process, single-thread, single-loop** ReAct agent. There is no multi-agent topology, no broker, no IPC by default. The system "graph" is a stack of objects in one Node process:

```
                +----------------------------------------------+
 user typing -> | Interactive TUI (pi-tui)                     |
                |   editor + tree view + tool renderers        |
                +----------------------------------------------+
                                  |  (calls)
                                  v
                +----------------------------------------------+
                | AgentSession (coding-agent)                  |
                |   sessions, compaction, slash commands,      |
                |   extensions, retry, bash execution          |
                +----------------------------------------------+
                                  |  (subscribes to events,
                                  |   calls .prompt/.steer/.followUp)
                                  v
                +----------------------------------------------+
                | Agent (pi-agent-core)                        |
                |   transcript, queues, runAgentLoop()         |
                +----------------------------------------------+
                                  |
                                  v
                +----------------------------------------------+
                | runLoop (agent-loop.ts)  - the ReAct loop    |
                |   stream assistant -> exec tools -> repeat   |
                +----------------------------------------------+
                                  |
                                  v
                +----------------------------------------------+
                | streamSimple (pi-ai)                         |
                |   provider stream w/ prompt caching          |
                +----------------------------------------------+
```

The actual ReAct loop lives in `packages/agent/src/agent-loop.ts:155-234` (`runLoop`). Two nested `while` loops:

```
// agent-loop.ts:163-231 (paraphrased shape)
while (true) {                                           // outer: drains follow-up queue
    while (hasMoreToolCalls || pendingMessages.length>0) { // inner: react cycle
        emit turn_start
        inject any pending steering messages
        message = streamAssistantResponse(...)
        if stopReason in (error, aborted): break
        toolCalls = filter(message.content, type==="toolCall")
        if toolCalls: executeToolCalls(...) -> push toolResults into context
        emit turn_end
        pendingMessages = await config.getSteeringMessages()
    }
    followUp = await config.getFollowUpMessages()
    if (followUp.length): pendingMessages = followUp; continue
    break
}
emit agent_end
```

That is the entire loop. There is no max-iterations cap, no stuck detector, no plan/critic split. The loop terminates when:
1. The assistant produces a message with no tool-calls, **and**
2. The steering queue is empty, **and**
3. The follow-up queue is empty.

Or earlier: error, abort, or every tool result in a batch sets `terminate: true`
(`agent-loop.ts:499-501`, `shouldTerminateToolBatch`).

There is **no hierarchical / multi-agent topology in the core**. The README spells this out:

> "**No sub-agents.** There's many ways to do this. Spawn pi instances via tmux, or build your own with extensions, or install a package that does it your way." (`packages/coding-agent/README.md:456`)

Sub-agents, when they exist at all, are an *example extension* (`packages/coding-agent/examples/extensions/subagent/`) that **`spawn()`s a separate `pi --mode json -p` subprocess** per delegation (`subagent/index.ts:265-310`):

```ts
// subagent/index.ts:265-267
const args: string[] = ["--mode", "json", "-p", "--no-session"];
if (agent.model) args.push("--model", agent.model);
if (agent.tools && agent.tools.length > 0) args.push("--tools", agent.tools.join(","));
```

Each sub-agent runs in its own OS process, its own context window, and the **only thing that crosses the boundary is JSON event lines on stdout** (`subagent/index.ts:313-348`: parse `message_end` and `tool_result_end` events, accumulate usage, return final assistant text). When the chain mode is used, the parent passes `{previous}` as a literal substring in the next agent's task prompt (`subagent/README.md:80-83`). There are no shared documents, no receipts, no dossiers. The artifact is a string.

## 4. Communication & handoff mechanism — KEY SECTION

This is the most important section for Handshake's question. **Pi has no broker and no documentation handoff at all.** Within a session, "handoff" means three things, in descending order of frequency:

### 4.1. Tool result -> next assistant turn (the only routine handoff)

This is the only boundary that *every* agent crosses. The sequence (from `agent-loop.ts:165-219`) is:

1. Assistant streams a message containing one or more `toolCall` content blocks.
2. `runLoop` calls `executeToolCalls`. Each tool's `execute(toolCallId, params, signal, onUpdate)` is invoked (`agent-loop.ts:577-594`).
3. The tool returns `{ content: (TextContent|ImageContent)[], details: any, terminate?: boolean }`. Only `content` is sent back to the LLM. `details` is for the UI/session log (`packages/agent/src/types.ts:291-302`).
4. A `ToolResultMessage` is constructed (`agent-loop.ts:668-678`) with `role: "toolResult"`, `toolCallId`, `toolName`, `content`, `details`, `isError`, `timestamp` and pushed into the transcript.
5. Loop iterates: the next `streamAssistantResponse` call passes the full transcript through `transformContext` -> `convertToLlm` -> provider stream.

Crucially: **no governance schema, no required envelope, no receipt**. Just `content` (returned to model) + `details` (kept locally). The contract for tool failures is also minimal — throw an `Error` and the loop converts it to a tool result with `isError: true` (`agent-loop.ts:597-604`):

```ts
} catch (error) {
    return {
        result: createErrorToolResult(error instanceof Error ? error.message : String(error)),
        isError: true,
    };
}
```

`createErrorToolResult` is one line: `{ content: [{ type: "text", text: message }], details: {} }` (`agent-loop.ts:651-656`). That's the entire error protocol.

### 4.2. User -> running agent ("steering" and "follow-up" queues)

Pi distinguishes two types of out-of-band messages with first-class queue primitives in `pi-agent-core`:

- **Steering** (`Enter` while agent works): injected *after the current assistant turn finishes its tool calls*, before the next LLM call (`agent-loop.ts:165, 218`).
- **Follow-up** (`Alt+Enter`): injected only when the agent would otherwise terminate (`agent-loop.ts:222-227`).

The implementation is `PendingMessageQueue` in `packages/agent/src/agent.ts:113-144`:

```ts
class PendingMessageQueue {
    constructor(public mode: "all" | "one-at-a-time") {}
    enqueue(m) { this.messages.push(m); }
    drain() {
        if (this.mode === "all") { ...return all... }
        // one-at-a-time: pop just the first
        const first = this.messages[0];
        this.messages = this.messages.slice(1);
        return [first];
    }
}
```

The loop calls `getSteeringMessages` / `getFollowUpMessages` callbacks (`agent-loop.ts:165, 222`); the wrapping `Agent` class hands them out from these queues. **Mode is configurable** so the user decides whether queued nudges are batched or trickled. This is a real handoff primitive: it preserves the loop, it is observable in the TUI, and it does not require any document to be in any shape.

### 4.3. Pi -> sub-pi (only via the optional subagent extension)

Spawn `pi --mode json -p`, parse JSONL events from stdout. Inputs are the system-prompt-append (a temp file), the tool allowlist (`--tools read,grep,find,ls`), and the literal task prompt as a CLI argument. The "handoff back" is a single string — the last assistant text — concatenated into the parent's tool result (`subagent/index.ts:284-291`). No structured schema, no checklist, no receipt.

### 4.4. Pi -> external system (RPC mode)

The other "handoff" is the `--mode rpc` protocol, used by external apps to drive pi headlessly. Defined in `packages/coding-agent/src/modes/rpc/rpc-types.ts`. It is **strict newline-delimited JSON over stdin/stdout**, explicitly warning against generic line readers:

> "RPC mode uses strict LF-delimited JSONL framing. Clients must split records on `\n` only. Do not use generic line readers like Node `readline`, which also split on Unicode separators inside JSON payloads." (`packages/coding-agent/README.md:444`)

Commands are typed unions (`rpc-types.ts:19-69`) with optional `id` for correlation: `prompt`, `steer`, `follow_up`, `abort`, `compact`, `set_model`, `bash`, etc. Responses echo the `id` and either `success: true` with optional `data` or `success: false` with `error`. **Events** (the actual transcript) are streamed alongside as `AgentSessionEvent` JSON lines.

Read this carefully: even when pi *is* hosted by an external coordinator (e.g. `openclaw/openclaw`), the coordinator does not send "packets" or "dossiers" — it sends `{type:"prompt", message:"..."}` and consumes events. The protocol is a **command-and-event tape**, not a document exchange.

### 4.5. What does NOT cross any boundary

No work-packet envelope, no receipt, no required acknowledgment, no dossier, no shared "this is what I did" schema. The session JSONL is the audit trail; the transcript is the only contract. The model sees user messages, assistant messages, tool-result messages, and (for reasoning models) thinking blocks. Nothing else.

## 5. Tool system

Tools are TypeBox-typed objects implementing `AgentTool` (`packages/agent/src/types.ts:308-331`):

```ts
export interface AgentTool<TParameters extends TSchema = TSchema, TDetails = any> extends Tool<TParameters> {
    label: string;
    prepareArguments?: (args: unknown) => Static<TParameters>;
    execute: (toolCallId, params, signal?, onUpdate?) => Promise<AgentToolResult<TDetails>>;
    executionMode?: "sequential" | "parallel";
}
```

`prepareArguments` is a pre-validation shim. The author uses it to fix model mistakes silently. Look at `edit.ts:90-114`:

```ts
function prepareEditArguments(input: unknown): EditToolInput {
    ...
    // Some models (Opus 4.6, GLM-5.1) send edits as a JSON string instead of an array
    if (typeof args.edits === "string") {
        try {
            const parsed = JSON.parse(args.edits);
            if (Array.isArray(parsed)) args.edits = parsed;
        } catch {}
    }
    // Legacy oldText/newText fields get folded into edits[]
    ...
}
```

That's a documentation-cost-zero way to absorb provider drift. Pi swallows it instead of propagating an error and burning a turn on repair. Compare to a governed system where a malformed envelope triggers a workflow retry.

### Built-in tool registry

`packages/coding-agent/src/core/tools/index.ts` exposes seven tools but **the default for `pi` is only four** (`system-prompt.ts:89`): `read`, `bash`, `edit`, `write`. `grep`, `find`, `ls` are opt-in. Nothing else.

### Edit tool — the cleverest one

`createEditTool` in `packages/coding-agent/src/core/tools/edit.ts` is the most opinionated. Its parameters require an `edits: [{oldText, newText}]` array (`edit.ts:31-51`). The description embeds the contract:

> "Each `edits[].oldText` is matched against the original file, not after earlier edits are applied. Do not emit overlapping or nested edits. Merge nearby changes into one edit." (`edit.ts:301-305`)

Three engineering tricks worth quoting:

1. **Fuzzy matching with progressive Unicode normalization** (`edit-diff.ts:34-55, 96-120`):

   ```ts
   export function normalizeForFuzzyMatch(text: string): string {
       return text.normalize("NFKC")
           .split("\n").map(l => l.trimEnd()).join("\n")  // strip trailing ws
           .replace(/[‘’‚‛]/g, "'")    // smart single quotes
           .replace(/[“”„‟]/g, '"')    // smart double quotes
           .replace(/[‐‑‒–—―−]/g, "-") // dashes
           .replace(/[  -   　]/g, " ");     // spaces
   }
   ```

   Exact match first, then fuzzy. This eats a *huge* class of repair-loop costs, because models routinely round-trip through smart quotes and en-dashes when copy-pasting.

2. **Line-ending and BOM round-trip preservation** (`edit.ts:367-381`): detect CRLF vs LF on read, normalize to LF for matching, restore on write, preserve BOM. The model never has to reason about it.

3. **File mutation queue** (`edit.ts:313, file-mutation-queue.ts`): every mutation through the same path is serialized so concurrent tool calls cannot race on the same file. A correctness fix that doesn't require the model to know.

### Tool result rendering vs LLM payload

`ToolDefinition` (the wrapped form in `tool-definition-wrapper.ts`) has separate `renderCall` and `renderResult` callbacks for the TUI, totally distinct from `execute`. The `details` field is the pivot — UI-only context that stays out of the model's context window. Pi can show a syntax-highlighted unified diff on screen while sending the model a one-line `"Successfully replaced N block(s) in foo.ts."` (`edit.ts:399`).

This is a critical asymmetry. Most "documentation as handoff" systems force the same artifact to satisfy both human review and model context.

### Bash tool

`createBashTool` (`tools/bash.ts:75-130`) spawns a detached child via `spawn(shell, [...args, command], { detached: true, stdio: ["ignore","pipe","pipe"] })`, streams stdout+stderr through `onData`, kills the entire process tree on abort (`killProcessTree`), supports an optional `timeout` arg, **with no default timeout** (`tools/bash.ts:35`). Truncation defaults are `DEFAULT_MAX_BYTES`/`DEFAULT_MAX_LINES` from `tools/truncate.ts`. There is no safelist, no policy, no permission popup. Containment is the user's job.

## 6. State & memory management

There is exactly one transcript and it is appended to in place. State management is layered:

### In-process state: `Agent.state`

Defined in `packages/agent/src/types.ts:264-289`. The transcript is a flat `AgentMessage[]`. Setters copy the top-level array but not deeply; mutating a returned array mutates the agent.

### Persistent session: JSONL tree

The coding-agent persists every emitted event to `~/.pi/agent/sessions/--<cwd>--/<ts>_<uuid>.jsonl`. Format is documented in `docs/session.md`. Each entry has `id` and `parentId`, forming a **tree**, which lets `/tree`, `/fork`, `/clone` navigate alternate branches of the *same* session in place. Version 3 of the format unified custom messages.

### Compaction

Implementation: `packages/coding-agent/src/core/compaction/compaction.ts`. Defaults (`compaction.ts:121-125`): `enabled: true, reserveTokens: 16384, keepRecentTokens: 20000`. The trigger logic in `agent-session.ts:1755-1833` (`_checkCompaction`) handles two cases explicitly:

1. **Overflow recovery**: provider returned a context-overflow error. Pi removes the failed assistant message from in-memory state (it stays in the session file), runs compaction, then auto-retries (`agent-session.ts:1782-1804`). Only one such recovery attempt per error.
2. **Threshold**: total context > `contextWindow - reserveTokens`. Pi compacts but does NOT auto-retry; the user continues manually.

The summarization prompt is hard-coded and small (`compaction.ts:454-485`). It uses a strict markdown skeleton — Goal / Constraints / Progress {Done/InProgress/Blocked} / Key Decisions / Next Steps / Critical Context — and explicitly tells the model to "preserve exact file paths, function names, and error messages." There's also an `UPDATE_SUMMARIZATION_PROMPT` (`compaction.ts:487-524`) used when a previous summary already exists, so iterative compactions don't lose accumulated context.

The cut-point selector (`findCutPoint`, `compaction.ts:386-448`) walks backwards from newest, accumulating a `chars/4` token estimate, and only ever cuts at user/assistant/custom/bashExecution boundaries — never at a tool-result (`compaction.ts:299-336`). Tool results follow their tool calls and are kept as a unit.

Compaction is lossy *in the active branch*; the full pre-compaction history is still in the JSONL file and can be revisited via `/tree`.

### Provider prompt caching

`packages/ai/src/providers/anthropic.ts:1090-1112`: Anthropic provider tags the last user message block with `cache_control: ephemeral` when caching is enabled, plus the last tool definition (`anthropic.ts:1141`). Combined with a stable transcript prefix, this means each turn re-uses the prior turn's cache. There is also a long-cache mode triggered by env var `PI_CACHE_RETENTION=long` (1 h on Anthropic, 24 h on OpenAI) per the README.

### Memory beyond a session

There is no built-in cross-session memory primitive. `AGENTS.md` / `CLAUDE.md` files are loaded at startup from cwd-up walking + global `~/.pi/agent/AGENTS.md` (`resource-loader.ts:59`) and concatenated into the system prompt. That is the entire "persistent memory" mechanism. Anything more elaborate (RAG, vector store, project DB) is an extension.

`pi-mom` (the Slack bot) demonstrates a simple bolt-on: it reads `MEMORY.md` at workspace and channel level on each run and prepends the contents (`packages/mom/src/agent.ts:69-103`). Pure file-on-disk, no schema.

## 7. Risk mitigation & failure handling

What you'd expect from a "minimal" harness:

- **No max-iteration cap on the ReAct loop.** None. The loop runs until the assistant stops calling tools or the user aborts.
- **Stuck detection: none built-in.** The user is the watchdog; pi shows live tool execution in the TUI.
- **Permission gates: none built-in.** The README explicitly rejects them: "**No permission popups.** Run in a container, or build your own confirmation flow with extensions inline with your environment and security requirements" (`coding-agent/README.md:458`). Extension hooks `beforeToolCall` (`packages/agent/src/types.ts:209`) and `afterToolCall` (`packages/agent/src/types.ts:223`) make permission gating a one-extension job; example extensions `permission-gate.ts`, `protected-paths.ts`, `confirm-destructive.ts`, `dirty-repo-guard.ts` ship in `examples/extensions/`.
- **Auto-retry for transient errors** (`agent-session.ts:2400-2413`): a regex matches `overloaded|provider returned error|rate.?limit|429|500|502|503|504|service.?unavailable|...|fetch failed|...|timeout|terminated|retry delay`. Auto-retry resets on the first successful assistant message (`agent-session.ts:554-562`). Context overflow is excluded — that path goes through compaction.
- **Malformed tool args**: `validateToolArguments` (in `pi-ai`) throws on schema mismatch; the catch in `prepareToolCall` (`agent-loop.ts:560-566`) converts the throw into an immediate error tool-result so the loop continues. Combined with `prepareArguments` shims (e.g. `prepareEditArguments`), the harness *prefers silent repair over loud rejection* whenever it can be made deterministic.
- **Abort path**: `Agent.abort()` aborts the underlying `AbortController`; tool `execute` functions are expected to listen on `signal`. `bash` kills the entire process tree (`bash.ts:104-106`); `edit` removes the listener and rejects (`edit.ts:329-335`).
- **Tool-batch early stop**: `terminate: true` on every result in a batch ends the loop without another LLM call (`agent-loop.ts:499-501`). Cheap manual brake without a "done" signal.

There is no equivalent of a typed "verdict" or "validator pass/fail." The closest substitute: an extension can return `{block: true, reason: "..."}` from `beforeToolCall` (`agent-loop.ts:546-553`), which becomes an error tool result the model reads.

## 8. Clever engineering tricks

A short list of what is genuinely thoughtful:

1. **Three-layer message model** (`agent/README.md:36-53`):
   ```
   AgentMessage[] -> transformContext() -> AgentMessage[] -> convertToLlm() -> Message[] -> LLM
                       (optional)                              (required)
   ```
   `AgentMessage` is `Message | CustomAgentMessages[keyof CustomAgentMessages]` (`types.ts:256`). Apps can declaration-merge new message types (notifications, artifacts, bash-execution rows) that show in the UI and get persisted, but get filtered out before the LLM call. **This is how pi keeps UI noise out of the model's context.** Compare to Handshake, where every artifact tends to round-trip through the prompt.

2. **`details` field on tool results** (`types.ts:292-302`). The split between `content` (model-visible) and `details` (UI/log only) is rigorous and used everywhere. `read` returns the file body in `content` but the truncation report only in `details`; `edit` returns `"Successfully replaced N block(s)"` in `content` but the unified diff only in `details`; `bash` returns the captured output in `content` but the temp-file path only in `details`. The model does not pay tokens for renderable metadata.

3. **`prepareArguments` shim per tool** (`types.ts:315`, `edit.ts:90-114`). Absorbs known model misbehaviour deterministically.

4. **Steering and follow-up queues are first-class loop primitives**, not application bolt-ons. The user can mid-flight redirect without aborting and without inventing a "control message" envelope (`agent-loop.ts:218`, `agent.ts:113-144`).

5. **Tree-structured session files** (`docs/session.md:3`). Branching in place via `parentId` in JSONL, no copy. Forking, cloning, and `/tree` all operate on the same file. This makes "what if I had asked a different question 20 messages ago" cheap.

6. **`afterToolCall` post-processor** (`agent-loop.ts:617-642`): can override `content`, `details`, `isError`, or `terminate`. Combined with `beforeToolCall`, this is the entire policy hook surface — and it's enough to implement permission gates, audit logging, content rewriting, conditional termination.

7. **Compaction is a pure function over messages, not a service** (`compaction.ts:530-590`). Settings control when, but the actual cut + summarize is one function call to the same provider with the strict markdown template. No separate "compactor agent" with its own context.

8. **No MCP. CLI tools + READMEs as the integration story.** From `coding-agent/README.md:454`: "**No MCP.** Build CLI tools with READMEs (see Skills), or build an extension that adds MCP support." Skills are markdown files with YAML frontmatter that the model reads on demand via the `read` tool. Per Zechner's blog: a Playwright MCP costs 13.7k tokens upfront on every session (`https://mariozechner.at/posts/2025-11-02-what-if-you-dont-need-mcp/`); a 225-token README costs only when the model reads it.

## 9. Token / cost behaviour

What pi does to keep token cost down:

- **Tiny system prompt** (`system-prompt.ts:131-147`). The default body is ~10 lines plus the four-tool list. Per Zechner, "system prompt + tool definitions total under 1,000 tokens — roughly 10x smaller than Claude Code's" (blog, 2025-11-30).
- **Tools list filtered to those actually selected** (`system-prompt.ts:89-92`). The prompt only mentions tools that have a one-line snippet. Disabled tools cost zero.
- **Skills are not preloaded into the prompt.** They live as files; the system prompt only mentions their existence (`formatSkillsForPrompt`). The model `read`s them when relevant.
- **Provider prompt caching** with stable prefixes; `cache_control` is appended to the last user/tool block on Anthropic, last tool def for tool definitions (see §6).
- **`details` channel keeps UI metadata out of the LLM** (see §8.2).
- **Compaction with iterative summary updates** so the summary itself doesn't grow unboundedly (`compaction.ts:487-524`).
- **`chars/4` heuristic for cut-point estimation** (`compaction.ts:232-290`) — conservative, deliberately overestimates.
- **Image content is estimated as 4800 chars / 1200 tokens flat** (`compaction.ts:272`). Crude but cheap.

What pi does NOT do:

- No automatic prompt summarization mid-turn.
- No vector retrieval.
- No "scratchpad" file. The user can choose to use a `TODO.md` or `PLAN.md` (per blog), but pi has no opinion.
- No global token budget. Compaction triggers on context-window pressure, not on a cost ceiling.

## 10. Comparison to ACP-style brokers

A direct comparison to Handshake's broker pattern:

| Concern | ACP broker (Handshake) | Pi |
|---|---|---|
| Number of model sessions | Many, per role | One |
| Number of processes | One per session, brokered via TCP/WebSocket | One |
| Inter-session protocol | ACP relay frames | None — there is no inter-session |
| Handoff envelope | Work Packets, dossiers, receipts | Tool result `{content, details}` only |
| Required artifact shape | Yes, governance-defined | No, only what the model emits |
| Stuck detection | Required at broker level | None — user watches |
| Validator role | Separate model session | An extension hook (`beforeToolCall`) or a sub-pi spawn |
| Audit trail | Separate logs across role sessions | Single JSONL transcript |
| Steering | Side-channel through broker | First-class queue primitive in the loop |
| Subagents | Core concept | Extension that spawns `pi --mode json` |

**Does pi need a broker? No, and it goes out of its way to argue you don't either.** The design hypothesis is that one well-instrumented loop with steering and follow-up primitives, plus on-demand sub-process delegation through `bash`, replaces almost every reason to build a broker.

What pi gives up by having no broker:
- Multiple concurrently-running model sessions on the same conversation.
- Cross-role isolation of context windows (must spawn a subprocess to get it).
- Independent role memories.
- A standardized "this is what happened" handoff format.

What pi gains:
- Zero cycles spent on inter-session protocol repair.
- The human can see *every token* on screen.
- The session JSONL is the audit trail; nothing else needs to be.
- Steering and follow-up are loop primitives, not protocol additions.
- Adding a "role" is `--append-system-prompt` + `--tools <list>`, not a new role definition with a packet schema.

The `pi-mom` Slack bot is the closest pi-mono ships to a brokered system: each Slack channel gets its own `AgentSession`, persistent in `data/<channel-id>/`. But there is no protocol between channels; they are independent agents talking to the same Slack workspace. The "broker" is Slack itself.

## 11. Lessons applicable to Handshake

Three specific lessons, in priority order:

**(a) Most "handoffs" don't need a document; they need a transcript channel.** Pi's central architectural bet is that `tool result -> next assistant turn` is the only handoff that has to exist, and that the transcript file is the only audit artifact you need. Handshake's pain — models spending cycles repairing receipts and dossiers — comes from forcing every cross-session handoff to satisfy *both* governance audit *and* model context. Pi splits these: `details` is for audit/UI, `content` is for model. If Handshake adopted just one primitive, it should be this split. Receipts could become `details`-equivalent rows in a session log, never round-tripped through any model's context.

**(b) Steering and follow-up belong in the loop, not in the protocol.** Pi proves you can mid-flight inject user input *or coordinator input* without aborting, without re-launching a session, and without a packet envelope. Two callbacks (`getSteeringMessages`, `getFollowUpMessages`) on a config object plus two queues with a `mode` switch (`one-at-a-time` vs `all`) covers the entire "the orchestrator wants to nudge mid-task" use case. Handshake's orchestrator could expose this as the *primary* steering mechanism for managed coder sessions — the broker drops a message into the queue, the next turn picks it up, no document required. This eliminates a class of "is the WP packet up to date?" loops because the steering message *is* the packet.

**(c) `beforeToolCall` / `afterToolCall` hooks replace most validator code.** Pi's two hook signatures (`packages/agent/src/types.ts:75-101`) collapse plan-mode, permission gates, content rewriting, and audit logging into one before-after pair on tool calls. A WP_VALIDATOR per microtask in Handshake could be re-cast as a synchronous `afterToolCall` hook on the *coder* session: it reads the tool result locally, decides pass/fail, and either annotates `details`, sets `isError`, or sets `terminate`. No second model session, no packet. If the validator does need a model — keep that — but invoke it inline via `completeSimple()` from the hook, the way pi's compaction uses the same provider for summarization (`compaction.ts:574-578`). The orchestrator stops mediating documents between coder and validator; the validator runs *inside* the coder turn.

A fourth, more speculative lesson: **`prepareArguments` shims as a governance pattern**. Whenever a model is observed to malform an artifact in a known, deterministic way, the right fix in pi-style is a one-function shim that absorbs it silently — not a workflow loop that repairs the document over multiple turns. Handshake's "WP packet not in correct shape" bottleneck would shrink dramatically if every known failure mode had a corresponding `prepareArguments`-equivalent normalizer running before the validator ever sees the packet.

## NOTES FOR SYNTHESIZER

1. **The single biggest contrast with ACP-broker patterns is that pi has no inter-session protocol at all.** Tool result `content` is the model handoff; tool result `details` is the audit handoff. They are stored together in one JSONL file and never combined. Any harness suffering from "documentation as handoff blocker" should consider whether its receipts are trying to be both at once.

2. **Steering and follow-up are loop primitives in `pi-agent-core`, not protocol additions.** Two callback hooks on the loop config + two `PendingMessageQueue` instances with a `mode` setting. This single design choice obviates an entire class of "how do we nudge a running session" broker code.

3. **`prepareArguments`, fuzzy match with Unicode normalization, file-mutation queue, and the `details`/`content` split** are all examples of *deterministic absorption of known failure modes* rather than *protocol-level repair loops*. This is the philosophical core of pi's low token burn.

4. **Sub-agents in pi are `spawn("pi", ["--mode","json","-p", ...])` — a subprocess with isolated context, returning a single string via JSONL parsing.** No persistent broker, no role registry. If Handshake needs role isolation, the cheapest implementation is process spawning, not a relay protocol.

5. **Pi has no max-iteration cap, no stuck detector, no built-in validator, no permission gate, no plan mode.** The README defends each absence individually. The cumulative claim is that one observable loop + extension hooks (`beforeToolCall`/`afterToolCall`) covers nearly everything a multi-agent broker tries to do, at vastly lower coordination cost. The extension surface is the lever.
